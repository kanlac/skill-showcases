## 拆解

如何实现的？

关键词：美学 aesthetic

沉淀了《前端美学：提示词指南》中的内容
https://github.com/anthropics/claude-cookbooks/blob/main/coding/prompting_for_frontend_aesthetics.ipynb

给了三组例子，没有美学提示，和有美学提示，最后生成的效果。
【图片】

可以看到，如果没有指导，Claude 通常会默认生成以白色和紫色为背景的简约设计。有了美学提示后，它会生成更多样化、更具视觉吸引力的设计。

大模型虽然有渊博的知识，精通设计原则、字体排印和色彩理论等，但如果不明确要求，它的这些能力都不会得到展现，而你也只能得到中规中矩的结果。
为什么会这样？是模型还不够智能吗？
当我说帮我生成一个好看的页面的时候，它就不能聪明一点，自动展开美学和前端设计相关的知识吗？
理论上大模型应该能具备这样的能力，但是为什么实际上没有？
一方面，token成本需要考虑。模型厂商不可能无限展开用户请求中的每个关键词，这不划算也没必要。
当我们寻求通过 agent 应用来获得比模型直出更好的结果时，本质上是在利用其在垂直领域沉淀的工作流、领域知识和最佳实践。
另一方面，当用户在说「好看」「精美」时，他并不知道自己在说什么，他描述的是一种模糊的感觉。含义需要澄清，意图需要校准。
这也是我做这个系列的初衷——我相信用户没有必要去学习比较那么多 agent/skill，
想象一下，当你看到一个案例，第一时间就看到效果，不是夸耀的宣传语，不是含义宽泛的功能介绍，就是真实的评测。
你立马就知道，「我就想要这个类似的」，或者「哦，这个跟我要做的东西还有点差别」。
你在查看 showcase 的过程，本质上也是在实行一次潜移默化的意图校准。

## 如何使用 

如果可以，请描述清楚用途、受众或技术限制等背景信息。
比如，相比于「帮我设计一个卖运动鞋的手机端商品卡片」，更好的说法是
```
请设计一个用于‘限量版球鞋抽签发售’APP 的移动端商品卡片。受众是追求潮流的 Z 世代用户，他们通常在几秒钟内做出购买决策，因此视觉重心必须是高清的大尺寸鞋子图片和倒计时组件，弱化文字描述。
```

若页面较复杂，可以和 webapp-testing 搭配使用，确保页面布局正常，功能可用。

## 边界与局限

frontend-skill 能用来出很好看的前端页面，但它能胜任所有前端开发工作吗？

主要侧重 UI 设计，建议利用它充分做好原型打磨，重新设计页面也可以，但不要让它做比较重的业务逻辑。如果你在处理状态管理或复杂的业务逻辑，可以使用拥有具体技术栈的开发经验或架构经验的 skill。（敬请期待后续评测）
