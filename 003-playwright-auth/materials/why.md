# 为什么需要 Playwright Auth Manager？

## 问题：AI 不会"记住"登录状态

想象一下，你让 AI 帮你从公司内部系统抓取数据，或者测试一个需要登录的功能。AI 打开浏览器，但它看到的是登录页面——它不知道你的账号密码，也不能像你一样"记住"登录状态。

每次都要你手动登录？太麻烦了。

## 解决方案：保存认证状态

### 什么是认证状态？

当你登录一个网站后，浏览器会保存一些信息来"记住"你：
- **Cookies**：网站发给浏览器的小数据片段，里面通常包含 session ID
- **LocalStorage**：网站在你浏览器里存的一些数据，可能包含 token

这些信息就是"认证状态"。有了它们，浏览器就知道"这是已登录的用户"。

### Playwright Auth Manager 做了什么？

简单说就三步：

1. **你登录一次**：运行脚本，手动在浏览器里登录
2. **保存认证信息**：脚本把 cookies 和 localStorage 保存到一个 JSON 文件
3. **AI 复用**：下次 AI 操作浏览器时，加载这个 JSON 文件，就自动处于"已登录"状态了

就像你把家里的钥匙（认证信息）复制了一把，交给了 AI 助手。它就能代替你进门了。

## 为什么手动复制 Cookie 不够用？

你可能想：我自己从浏览器复制 Cookie 粘贴到代码里不就行了？

问题是：
1. **不只是 Cookie**：很多现代网站还用 localStorage 存 token
2. **容易出错**：手动复制粘贴，格式容易错，过期时间也难处理
3. **不够自动化**：每次过期都要重新复制
4. **AI 不友好**：你要告诉 AI 怎么用这些 Cookie，它还不一定理解

Playwright Auth Manager 把这一切自动化了。

## 为什么只能本地使用？

因为这个 JSON 文件里包含的是**你的真实登录凭证**。

想象一下：
- 你的 session cookie 就像你的门禁卡
- 你的 auth token 就像你的密码

如果把这个 JSON 文件放到云端服务器、提交到 Git、或者分享给别人，就相当于把你的钥匙随便乱放——非常危险。

所以这个 Skill 的设计原则就是：
- ✅ 本地开发调试：安全，可控
- ✅ 团队内部分享：需要时可以加密传输
- ❌ 云端部署：绝对不行
- ❌ 公开仓库：自动检查 .gitignore

## 核心价值：一次配置，持续使用

没有这个 Skill，你需要：
1. 自己写代码管理 cookies
2. 手动处理 localStorage
3. 处理过期和刷新逻辑
4. 告诉 AI 怎么用这些认证信息

有了这个 Skill，你只需要：
1. 运行脚本，登录一次
2. 配置 MCP Server
3. AI 自动处于登录状态

就这么简单。
