# 创作思考

## 为什么做这个 Skill

### 初衷：调试太麻烦了

在本地开发需要登录的功能时，经常遇到这些场景：
- 测试不同用户看到的界面是否正确
- 检查不同权限用户的展示效果
- 验证多账号场景下的数据隔离

每次都要：
1. 手动登出当前账号
2. 登录另一个测试账号
3. 测试功能
4. 又要切回来……

太繁琐了。尤其是有些网站登录流程很复杂（两步验证、验证码等），切换一次账号就要折腾好几分钟。

### 解决方案

做了 Playwright Auth Manager 之后：
- 提前保存好几个测试账号的认证状态
- 配置多个 MCP 实例
- 让 Agent 代替我切换账号、检查不同用户的登录态和展示态
- 我只需要看结果

效率提升太明显了。

## 为什么现在是发布的好时机

### 市场空白

我调研了一圈，发现：
- **没有现成的 Skill** 做这个事情
- Playwright MCP 是刚发布不久的新东西
- 大家对 MCP + 认证管理的需求很强烈
- 但目前都是自己手写脚本，没有标准化方案

### 这个 Skill 的价值

**让用户不需要关心实现细节**。

如果没有这个 Skill，你要：
1. 了解 Playwright 的 Storage State API
2. 写脚本保存认证状态
3. 配置 MCP Server 参数
4. 处理多会话的复杂配置
5. 记得加 .gitignore

有了这个 Skill，Agent 会：
1. 自动引导你运行正确的命令
2. 帮你配置 MCP Server
3. 提醒你检查 .gitignore
4. 验证认证是否生效

**一次性配置好，后续完全自动化**。

### 适用场景

这个 Skill 真正适合的场景：
- ✅ 本地开发调试需要登录的功能
- ✅ 数据抓取（如公司内部系统、私有项目）
- ✅ 多账号测试（权限、展示态等）
- ✅ 自动化测试（模拟已登录用户）

不适合的场景：
- ❌ 生产环境（安全风险）
- ❌ 云端服务器（凭证泄露）
- ❌ 公开项目（敏感数据）

## 与竞品对比

### 调研结果

目前市面上**没有专门的 Skill 做这个事情**。

现有方案的问题：
1. **手动复制 Cookie**：
   - 只能复制 Cookie，拿不到 localStorage
   - 容易出错（格式、过期时间）
   - 不够自动化

2. **手写 Playwright 脚本**：
   - 需要自己学习 API
   - 每个项目都要重新写
   - 没有标准化流程

3. **浏览器插件**（如 EditThisCookie）：
   - 无法与 Coding Agent 集成
   - 需要手动导入导出
   - 不支持 MCP

### Playwright Auth Manager 的优势

- **一次配置，持续使用**：运行脚本保存一次，后续自动复用
- **Agent 友好**：直接集成到 MCP，Agent 自动知道怎么用
- **多会话支持**：轻松配置多个账号，无缝切换
- **安全设计**：自动检查 .gitignore，防止泄露

## 写作思路

### 文章定位

这是第一个自研 Skill，有几点特殊性：
1. **技术性更强**：需要平衡技术深度和可读性
2. **受众更垂直**：主要面向开发者
3. **填补空白**：市场上还没有类似方案

### 写作重点

1. **痛点要清晰**：开发调试时切换账号很麻烦
2. **解决方案要简单**：保存认证 JSON，Agent 复用
3. **价值要明确**：一次配置，持续使用，不需要关心实现
4. **边界要清楚**：仅限本地使用，强调安全
